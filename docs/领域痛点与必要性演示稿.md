# KnowFlow 领域痛点与开发必要性演示稿

各位老师好，我的项目是一个面向“大数据专业课程学习”的 RAG 知识库问答系统，名字叫 KnowFlow，中文名叫“知汇”。

我想先不讲系统功能，先讲为什么要做它。

我们大数据专业的学习有一个很真实的问题：资料非常多，但知识很难真正“流动”起来。

比如我们学机器学习、数据挖掘、Spark、Hadoop、数据库、信息系统分析与设计这些课程时，资料来源特别分散。有课件 PDF，有老师发的 Word 文档，有实验指导书，有教材截图，有网上查到的博客，还有自己整理的笔记。看起来资料很多，但真正遇到问题时，经常还是不知道答案在哪。

举个很常见的场景：我想复习“决策树的信息增益怎么算”，或者想知道“RAG 系统为什么要做文本分块”，我可能要去翻 PPT、翻 PDF、翻老师给的文档，甚至还要全局搜索关键词。问题是，关键词不一定能搜到，因为我问的是一个理解型问题，不一定和文档里的原话完全一样。

所以这里的痛点不是“没有资料”，而是“资料太碎、太散、太难快速转化成可用答案”。

现在已有的方式主要有几种。

第一种是人工翻资料。这个方式最可靠，但效率很低。尤其期末复习、做课程设计、写实验报告的时候，时间很紧，人工查找会占掉很多精力。而且人工方式很依赖个人整理能力，如果平时没有做好笔记，后面查起来就很痛苦。

第二种是直接问通用大模型，比如通义千问、ChatGPT。它确实能回答很多问题，但问题在于，它不一定知道我们课程里老师具体讲了什么，也不知道我们上传的那份资料里是怎么定义的。它给出的答案可能很通用，甚至有时候和课程要求不一致。对于课程作业来说，我更需要的是“基于我自己的课程资料来回答”，而不是一个泛泛的互联网答案。

第三种是用网盘搜索、PDF 搜索、文档关键词搜索。这种方式只能解决“找原文”的问题，不能很好解决“理解问题”。比如我问“为什么要做向量化入库”，文档里可能写的是“embedding 后写入 Chroma”，关键词完全不一样，普通搜索就很容易漏掉。

所以我认为这里真正的需求缺口是：学生需要一个能够理解问题、又能回到课程资料本身找依据的学习助手。

也就是说，它不能只是聊天机器人，也不能只是文件搜索工具。它应该介于两者之间：既能像大模型一样理解自然语言问题，又能像知识库一样把答案限制在我上传的专业资料范围内，并且告诉我答案来源于哪份文档、哪个片段。

这就是我选择 RAG 的原因。

RAG 不是为了炫技术，而是刚好对应这个问题：大模型负责理解和生成，知识库负责提供可靠依据。这样系统回答的时候，不是凭空编，而是先从课程资料里检索相关内容，再组织成答案。

我这个系统虽然在形式上参考了市面上的知识库问答产品，比如企业知识库、智能客服、文档问答系统，但它存在的理由并不是简单模仿。因为这些产品大多面向企业办公，强调客服、制度查询、团队协作；而我的系统面向的是大数据专业学生的课程学习和项目实践。

这个场景有几个特点：

第一，知识更新快。大数据领域技术栈变化很快，课程资料、实验环境、工具版本经常变，固定教材不一定够用。

第二，知识链条长。比如一个 RAG 系统本身就涉及文档解析、文本分块、embedding、向量数据库、检索、LLM 调用、前后端交互。学生不是只想查一个定义，而是想理解整个工程流程。

第三，学习资料半结构化严重。PDF、PPT、Word、Markdown、实验报告混在一起，传统搜索体验很差。

第四，学生需要可解释答案。尤其做作业和答辩时，不能只给一个答案，还要知道这个答案从哪里来，能不能回到资料中验证。

所以这个系统的必要性在于：它把“资料管理”“语义检索”和“问答解释”连接起来，让学生可以围绕自己的课程资料建立一个可持续积累的专业知识库。

对我个人来说，这个项目也不只是做一个页面或者调一个 API。它能体现一个比较完整的大数据工程实践流程：文档采集、解析、分块、向量化、入库、检索、生成、引用溯源、前后端交互、系统状态监控。这些都和我的专业方向比较贴合，也能作为简历里比较完整的工程项目。

总结一下，这个系统要解决的不是“让 AI 回答问题”这么简单，而是解决大数据专业学习中资料分散、查找低效、通用大模型不贴合课程资料、人工整理成本高的问题。它的价值在于让已有资料真正变成一个可以被提问、被检索、被复用的知识系统。

这也是我开发 KnowFlow 的主要原因。
