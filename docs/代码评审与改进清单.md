## 一、核心能力（RAG 本身）

### 🔴 P0-1 用真正的语义 embedding 替换哈希词袋

- **出处**：`backend/app/services/embedding_service.py:8` `HashingEmbeddingService`
- **问题本质**：当前实现是把 token 哈希进 384 维后 ±1 累加，是**特征哈希 / 词袋模型**，不是语义 embedding。中文靠单字 + 二元组匹配，本质是字面 n-gram 重叠打分。这与 README 第 17 行「能理解自然语言问题，而不是只匹配关键词」的核心主张**直接矛盾**——现状恰恰就是关键词匹配，只是套了 Chroma + 余弦相似度的壳。换个说法提问（如「怎么切分文本」→「如何把长文档拆小」）就会检索失败。
- **改进方案**：
  - 接入 DashScope `text-embedding-v3`（或 `bge-large-zh`）。已在用的 OpenAI SDK 同一个客户端即可调用，约 20 行改动。
  - 保留 `HashingEmbeddingService` 作为无 key 时的降级/测试实现，通过配置开关切换（面向 service 边界已经预留好了，注释里也写了「replace with bge or DashScope later」）。
  - 注意 embedding 维度会从 384 变化，需要清空并重建 Chroma collection。
- **验收标准**：用与资料**不同措辞**的问题提问，仍能召回正确 chunk；检索得分对同义改写稳定。
- **备注**：这是全项目性价比最高的一处改动——把项目从「假的」变成「真的」。

### 🔴 P0-2 把会话历史真正喂进 LLM

- **出处**：`backend/app/services/rag_service.py:14` `answer()` 只接收当前 `question`，历史消息从不进入 prompt。
- **问题本质**：所谓「多轮会话」，每一轮都是失忆的单轮对话。数据库存了全量 `ChatMessage`，却只用于 token 求和统计（`admin.py`）。
- **改进方案**：
  - `answer()` 增加 `history: list[ChatMessage]` 入参，在 `chat.py` 调用前取出该 conversation 的历史消息。
  - 构造 `messages` 时把历史轮次按 role 拼进去（注意上下文长度预算，必要时做截断/滑窗）。
  - 检索 query 可考虑用「历史 + 当前问题」做指代消解后的改写 query（进阶）。
- **验收标准**：第二轮用「它」「上面那个」等指代能被正确理解并回答。

## 二、功能正确性（演戏 / Bug）

### 🟠 P1-1 前端「历史会话」只显示最后一条消息

- **出处**：`frontend/src/App.vue:302` `loadConversation()` 只取最后一条 assistant 消息塞进单个 `answer` ref。
- **问题本质**：没有消息流渲染，「历史会话」实际是一个「只能看最后一页」的日志查看器。
- **改进方案**：引入 `messages` 数组状态，按顺序渲染 user/assistant 气泡；点击历史会话加载完整对话。与 P0-2 配套。
- **验收标准**：打开任一历史会话可看到完整多轮往返。

### 🟠 P1-2 `totalChunks` 统计恒为 0（界面数字撒谎）

- **出处**：`frontend/src/App.vue:123`、`:1023`、`:1042`；对应 `backend/app/schemas/document.py`。
- **问题本质**：前端读 `doc.chunk_count`，但 `DocumentRead` schema **根本没有** `chunk_count` 字段，`doc.chunk_count || 0` 恒为 0。于是「总分块数」实际显示的是当前预览文档的 chunk 数，文档列表里每个文档永远显示「0 chunks」。界面统计在公开撒谎。
- **改进方案**：
  - 在 `DocumentRead` 增加 `chunk_count` 字段，后端查询时用 `func.count` 或关系聚合填充。
  - 前端 `totalChunks` 改为对真实字段求和，移除把 `chunks.value.length` 当初始值的错误写法。
- **验收标准**：列表每个文档显示正确 chunk 数；顶部总数 = 各文档之和。

### 🟠 P1-3 检索 trace 的 `top_k` 字段名撒谎

- **出处**：`backend/app/services/rag_service.py:88` `_build_retrieval_trace` 把 `top_k` 设为返回条数 `len(sources)`，而非请求参数。
- **问题本质**：命中不足 top_k 时字段值与语义不符，误导调参。
- **改进方案**：trace 里区分 `requested_top_k`（入参）与 `returned_count`（命中数）。
- **验收标准**：命中数 < 请求数时两个字段值不同且各自准确。

## 三、工程化（零证据的主张）

### 🟠 P1-4 补齐测试（当前零测试）

- **出处**：全仓库无 pytest、无 vitest，`frontend/package.json` 无 test script。
- **问题本质**：`docs/工程化推进计划书.md` 写了「可测试」，但没有一行测试。`text_chunker.py`、`document_parser.py` 是最适合单测的纯函数，一个都没有。
- **改进方案**：
  - 后端引入 `pytest`，优先覆盖：`split_text` 的边界（空文本、超长段落、overlap 校验）、`estimate_token_count`、`normalize_text`、各 parser 的最小样例。
  - 关键 API 用 FastAPI `TestClient` 做冒烟测试（创建 KB → 上传 → 提问）。
  - 前端至少给 `client.js` 和几个纯函数加 vitest。
- **验收标准**：`pytest` 通过且核心纯函数分支覆盖；CI 可跑。

### 🟠 P1-5 引入日志（当前零日志）

- **出处**：整个后端无一行 logger；`backend/app/api/documents.py:92` `except Exception` 把异常吞进 `error_message` 字段，服务端不留堆栈。
- **改进方案**：
  - 配置标准 `logging`（或 loguru），在文档处理、LLM 调用、向量读写关键路径打 info/error。
  - `except` 分支保留 `logger.exception(...)`，不要静默吞异常。
- **验收标准**：一次失败上传能在服务端日志看到完整堆栈与上下文。

- **出处**：`git log` 仅两个 commit（一次性倾倒 + 改 README）。
- **问题本质**：对课程/作品项目，提交历史就是工作过程证明；当前历史看起来像「一晚上生成的」。
- **改进方案**：此后按功能/修复小步提交，写清 commit message；本清单每完成一项就单独提交。
- **验收标准**：后续历史呈现可读的、按主题分组的演进。

## 四、后端硬伤

### 🟠 P1-6 文档处理阻塞事件循环

- **出处**：`backend/app/api/documents.py:29` `upload_document` 是 `async def`，但解析、分块、嵌入、Chroma 写入全是同步重活。
- **问题本质**：一个大 PPT 上传会阻塞整个事件循环，所有请求卡住。`status="processing"` 永远无法被客户端观测到（处理在请求内同步完成），该状态字段形同虚设。
- **改进方案**：
  - 短期：把重活丢进 `run_in_threadpool` / `asyncio.to_thread`，或改用同步路由函数（FastAPI 会自动线程池化）。
  - 正解：`BackgroundTasks` 或任务队列异步处理，上传立即返回 `processing`，前端轮询/推送状态。
- **验收标准**：上传大文件期间其他接口仍可正常响应；`processing → finished/failed` 状态可被前端观测。

### 🟠 P1-7 上传无大小限制，整体读入内存

- **出处**：`backend/app/api/documents.py:55` `content = await file.read()` 一次性吞入内存。
- **改进方案**：限制最大文件大小（配置项）；超限返回 413；大文件流式落盘。
- **验收标准**：超过阈值的上传被拒绝且不撑爆内存。

### 🟠 P1-8 SQLite 与 Chroma 双写无一致性

- **出处**：上传 `documents.py:85-96`（flush 后先写向量再 commit，失败留孤儿向量）；删除 `knowledge_bases.py:117` 与 `documents.py:153`（先删向量和物理文件再 commit，失败则文件没了记录还在）。
- **改进方案**：
  - 明确写入顺序与失败补偿；至少保证 DB 事务提交成功后再做不可逆的向量/文件操作，或反之并加清理任务。
  - 提供「重建索引」维护接口修复不一致（计划书里已列此目标）。
- **验收标准**：在任一步骤人为制造失败后，系统不残留孤儿向量/孤儿文件，或有工具可修复。

### 🟡 P2-2 CORS 配置无效组合

- **出处**：`backend/app/main.py:24` `allow_origins=["*"]` 与 `allow_credentials=True` 同时设置。
- **问题本质**：这是浏览器 CORS 规范里的无效组合，`*` 与凭证不能共存，属于「照抄教程没细想」。
- **改进方案**：明确列出允许的前端源（如 `http://127.0.0.1:5173`），或在不需要凭证时关掉 `allow_credentials`。
- **验收标准**：跨域请求行为符合预期，无浏览器告警。

### 🟡 P2-3 缺少流式输出

- **出处**：`rag_service.py` / `chat.py` 一次性返回全量回答；前端 axios 超时设 60s（`client.js:6`）。
- **问题本质**：2026 年的问答 UI 让用户干等整段回答，体验落后一个时代。
- **改进方案**：后端用 SSE / `StreamingResponse` 流式吐 token；前端增量渲染。
- **验收标准**：回答逐字/逐句出现，首字延迟显著低于整体完成时间。
