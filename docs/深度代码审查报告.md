# KnowFlow 深度代码审查与稳定性报告

审查日期：2026-07-21
审查分支：`codex/comprehensive-hardening`

## 1. 审查范围

本轮检查覆盖：

- FastAPI 的 7 组 API、SQLAlchemy 模型、Pydantic schema、RAG/LLM/embedding/vector store 服务。
- Vue 入口、全局状态、API 客户端、消息渲染、依赖与生产构建。
- SQLite、Chroma 和上传目录之间的一致性边界。
- 环境变量、CORS、日志、文件路径、输入校验和 prompt injection 风险。
- Python/npm 依赖漏洞、静态检查、测试覆盖、GitHub Actions 和 Git 工作区规则。

## 2. 总体结论

项目已经从“可演示 MVP”提升为“有自动验证、失败补偿和明确运行边界的本地 RAG 应用”。核心 RAG 链路现在使用真实语义 embedding，会话历史进入 LLM，统计字段与 trace 语义一致，上传不会整体吞入内存，关键纯函数与 API 有回归测试。

当前最适合的定位仍是**本地单用户课程/作品项目**，不是可直接公网部署的多租户产品。主要原因是尚无认证授权、限流、任务队列、数据库迁移框架和多进程共享任务状态。README 已明确这一边界。

## 3. 已修复问题

### 3.1 正确性

| 问题 | 原风险 | 处理结果 |
| --- | --- | --- |
| 哈希词袋被称为语义检索 | 同义改写召回失败 | 接入 DashScope `text-embedding-v3`；无 key 时明确使用 hashing 降级 |
| 多轮对话失忆 | “它/上一个”无法理解 | 历史按时间顺序进入 LLM；检索 query 拼接最近用户问题辅助指代消解 |
| Top K 设置不生效 | 请求 schema 默认值始终覆盖系统设置 | 请求改为可选值；未传时使用系统设置 |
| trace 字段语义错误 | `top_k` 实际是返回数量 | 拆为 `requested_top_k` 和 `returned_count` |
| chunk 总数错误 | 前端长期显示 0 或混入预览数量 | 后端聚合真实数量；前端纯函数求和 |
| 会话排序不更新 | 活跃会话仍停留在旧位置 | 每次成功问答更新 `conversation.updated_at` |
| 空文件/无文本文件被标为成功 | 形成无向量的 finished 文档 | 空上传返回 400；无法提取文本的文档标记 failed |
| 知识库删除遗留会话 | SQLite 中出现孤儿记录 | 显式删除消息、会话、chunk、文档后再删除知识库 |

### 3.2 稳定性与一致性

| 问题 | 原风险 | 处理结果 |
| --- | --- | --- |
| 上传整体读入内存 | 大文件内存峰值不可控 | 1 MiB 分块落盘；默认 50 MiB 上限；超限返回 413 |
| 同步解析阻塞事件循环 | 上传大文档时其他请求卡顿 | 解析、分块、embedding、Chroma 写入进入线程池 |
| SQLite/Chroma 双写窗口 | 失败后留下孤儿向量或错误 DB 状态 | 最终 DB commit 纳入 try；失败 rollback 并按 chunk id 补偿删除向量 |
| 删除先移除外部资源 | DB commit 失败后记录仍在但文件已丢 | DB 先提交删除；向量/文件做幂等清理并记录异常 |
| 模型切换复用 collection | embedding 维度不一致导致运行时错误 | collection 名绑定 provider/model；提供 KB 级重建索引 API |
| embedding 大批量单请求 | 上游输入限制和失败面过大 | 按配置分批请求；Chroma 按 100 条 upsert |
| 外部 API 无明确超时 | 请求可能长时间悬挂 | LLM/embedding 使用可配置 60 秒超时 |
| 历史上下文无预算 | 长会话挤爆上下文窗口 | 同时限制历史条数和总字符数，保留最新消息 |
| SQLite 外键默认关闭 | 约束声明不生效 | 每个 SQLite 连接执行 `PRAGMA foreign_keys=ON` |

### 3.3 安全与规则

| 问题 | 原风险 | 处理结果 |
| --- | --- | --- |
| RAG 资料可注入指令 | 文档内容劫持系统行为 | system prompt 明确把资料视为不可信数据并忽略其中指令 |
| 上游错误详情直出 | 泄露供应商返回内容 | 客户端只得到状态摘要，完整堆栈留在服务端日志 |
| 文件路径直接 unlink | 被污染的 DB 路径可能删除上传目录之外的文件 | 删除前验证路径必须位于 upload root |
| CORS `*` 与 credentials 并用 | 浏览器行为不一致 | 来源改为环境变量白名单，关闭 credentials，收窄方法和 header |
| 空白名称/问题通过校验 | 产生不可用实体 | schema 在长度校验前统一 trim 并拒绝空白值 |
| 依赖版本过旧 | Python 审计最初报告 46 条漏洞 | 升级 FastAPI/Starlette、multipart、pypdf、pytest；npm 0 漏洞 |
| 无仓库级换行/格式规则 | Windows/Linux diff 噪声 | 增加 `.editorconfig`、`.gitattributes` 和 Ruff 配置 |

## 4. 架构评价

### 4.1 做得好的地方

- 后端 `api / schemas / models / services` 分层清晰，领域边界容易继续演进。
- parser 和 chunker 是纯函数，适合测试；DOCX/PPTX 表格内容也被提取。
- 回答携带来源、分数与 trace，产品上保留了可验证性。
- SQLite 保存事实数据、Chroma 作为可重建索引，这一主从关系现在已在代码中落实。
- embedding 服务保留本地 deterministic fallback，使测试和无 key 开发可运行。

### 4.2 本轮结构调整

- `create_app()` 集中组装 FastAPI、中间件和路由，测试可覆盖真实应用。
- 消息流从 `App.vue` 拆到 scoped 组件；文档统计抽为纯函数并加入 Vitest。
- runtime 与 dev 依赖拆分为 `requirements.txt` / `requirements-dev.txt`。
- GitHub Actions 分别执行 Python 与 Node 测试、构建、lint 和依赖审计。

## 5. 测试与验证证据

- 后端：18 项 pytest，覆盖 parser、chunker、trace、指代检索 query、KB CRUD、重复名称、会话外键、Top K 设置、历史传递、级联清理、上传与大小限制。

## 6. 已知残余风险与建议顺序

### P1：部署前必须处理

1. **认证与授权**：所有管理 API 都是无认证的。当前必须只监听 `127.0.0.1`；公网部署前需要用户、会话、KB 所有权和管理员权限。
2. **后台任务**：上传虽然不阻塞事件循环，但请求仍等待解析完成。生产环境应使用任务队列，并让前端轮询 `processing → finished/failed`。
3. **数据库迁移**：`create_all` 不会升级已有表结构。开始保存重要数据前应引入 Alembic，并为 SQLite 使用 batch migration。
4. **限流与配额**：LLM、embedding、上传和搜索接口缺少按用户/来源限流，公网会产生费用与资源滥用风险。

### P2：下一轮架构优化

1. `App.vue` 仍承担知识库、设置、文档和聊天多域状态；建议继续拆为 route-level views 与 `useChat/useKnowledgeBases` composables。
2. Vite 主 JS chunk 约 1.14 MB，主要来自全量 Element Plus；建议使用按需组件导入和视图级动态加载。
3. 增加 SSE 流式回答，降低首字延迟；同时加入取消请求和重试策略。
4. 建立 RAG 评测集，记录 Recall@K、MRR、拒答准确率和同义改写稳定性；目前测试证明工程行为，不证明回答质量。
5. 对 PDF/DOCX/PPTX 增加压缩炸弹、页数/对象数和解析耗时限制；当前只有上传字节数限制。

## 7. 依赖审计说明

升级后有修复版本的 Python 漏洞均已消除。`chromadb==1.5.9` 仍对应 `PYSEC-2026-311`，上游没有可用修复版本。该公告涉及服务端鉴权边界；KnowFlow 使用本地进程内 `PersistentClient`，不启动或暴露 Chroma Server，因此当前风险被部署模型隔离。CI 对这一单项使用显式例外，其余漏洞仍会使流水线失败。

## 8. 最终判断

在“本地单用户课程 RAG 工具”的约束内，当前代码已经具备可靠的错误边界、回归测试、依赖审计和可恢复索引机制，可以安全地继续迭代和展示。若目标变为公网、多用户或承载重要资料，则第 6 节 P1 项不是可选优化，而是上线前置条件。
